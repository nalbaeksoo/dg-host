#### CreateDate : 2026-04-16

Execute to Parse count 가 음수가 나오는 경우
대표적인 예제를 몇 가지로 나눠볼게요.

### 1) 문법 오류 때문에 parse는 되지만 execute까지 못 가는 경우

애플리케이션이 동적 SQL을 만들다가 오타를 냈다고 해보죠.

```sql
SELECT * FORM employees WHERE employee_id = :1
```

여기서 `FORM` 오타 때문에 Oracle은 이 SQL을 **parse 시도**하다가 에러를 냅니다.  
그러면:

- `parse count`는 증가
    
- `execute count`는 증가하지 않음
    

이런 일이 반복되면 `% Execute to Parse count`가 확 떨어질 수 있습니다.

실제 앱 패턴은 이런 식입니다.

```java
for (id in ids) {
    sql = "SELECT * FORM employees WHERE employee_id = ?";
    prepare(sql);   // parse 시도
    execute();      // parse 에러로 여기까지 못 감
}
```

---

### 2) 애플리케이션이 매번 prepare만 하고 조건에 따라 execute를 안 하는 경우

예를 들어 앱이 먼저 SQL을 준비해두고, 어떤 조건이면 실행을 취소하는 로직이 있다고 해보겠습니다.

```java
PreparedStatement ps =
    conn.prepareStatement("DELETE FROM orders WHERE order_id = ?");  // parse

if (!approved) {
    return;   // execute 안 함
}

ps.setInt(1, orderId);
ps.executeUpdate();   // execute
```

이 코드가 대량으로 반복되면:

- parse는 많이 발생
    
- execute는 일부만 발생
    

하게 됩니다.

즉, “미리 준비만 해놓고 실제론 안 쏘는” 패턴이 parse 과다의 흔한 원인입니다.

---

### 3) ORM / 프레임워크가 존재 여부 확인용으로 SQL을 계속 생성하다가 중간에 포기하는 경우

예를 들어 화면 필터 조건을 조합하면서 SQL을 계속 만들어 `prepare`하지만,  
최종적으로 검색 조건이 비어 있으면 실행을 안 하는 경우입니다.

```pseudo
sql = buildSearchSql(filters)
stmt = prepare(sql)      -- parse

if filters.isEmpty():
    close(stmt)
    return               -- execute 안 함
```

화면 요청이 많고 이런 패턴이 반복되면 parse만 쌓입니다.

---

### 4) 동일 SQL을 매번 새로 parse하지만 예외 처리로 execute 전에 빠지는 경우

예를 들어 입력값 검증이 DB 호출 직전에 실패하는 구조입니다.

```java
PreparedStatement ps =
    conn.prepareStatement("INSERT INTO audit_log(user_id, action) VALUES (?, ?)");  // parse

if (action == null) {
    throw new IllegalArgumentException();
}

ps.setInt(1, userId);
ps.setString(2, action);
ps.executeUpdate();
```

여기서 개발자가 `prepareStatement()`를 너무 일찍 호출하면,  
유효성 실패 건마다 parse만 생기고 execute는 안 생깁니다.

---

### 5) DB 링크, 권한, object resolution 문제로 parse 단계에서 막히는 경우

예를 들어 참조 객체가 없거나 synonym resolution 문제, 권한 문제 등이 있으면  
실행 전에 parse 단계에서 실패할 수 있습니다.

```sql
SELECT * FROM missing_table WHERE id = :1
```

또는

```sql
SELECT * FROM hr.secret_table WHERE empno = :1
```

권한이 없으면 parse 시점에 에러가 나서 execute까지 가지 못할 수 있습니다.

이 경우도:

- parse count 증가
    
- execute count 증가 없음
    

입니다.

---

### 6) 커넥션 풀/미들티어가 statement를 테스트 용도로 prepare만 하는 경우

일부 비효율적인 코드나 프레임워크 버그에서는 실제 업무 SQL 전에  
statement 생성 가능 여부만 확인하려고 `prepare`를 남발하는 경우가 있습니다.

```java
PreparedStatement ps = conn.prepareStatement(sql);  // parse
ps.close();                                         // execute 없음
```

이게 반복되면 parse만 매우 많이 쌓일 수 있습니다.

---

### 숫자로 보면

예를 들어 1분 동안:

- 1,000번 `prepare/parse`
    
- 그중 실제 실행은 300번
    

이면

- `parse count total = 1000`
    
- `execute count = 300`
    

이고,

`% Execute to Parse count = (1 - 1000/300) * 100 = -233.3%`

처럼 음수가 크게 나올 수 있습니다.

---

### 실무에서 가장 흔한 원인

현장에서 자주 보는 건 대체로 이 순서예요.

1. **오류 SQL 반복**
    
2. **prepare를 너무 일찍 호출하는 애플리케이션 구조**
    
3. **동적 SQL 남발 + cursor 재사용 실패**
    
4. **프레임워크/ORM이 필요 이상으로 statement를 생성**
    
5. **짧은 측정 구간에서 parse/execute 시차 발생**
    

---

### AWR에서 같이 보면 좋은 것

이 상황이 의심되면 같이 봐야 하는 지표는 보통 이겁니다.

- `parse count (total)`
    
- `execute count`
    
- `parse count (failures)`
    
- `opened cursors current`
    
- `session cursor cache hits`
    
- `hard parses per sec`
    
- SQL ordered by Parse Calls
    
- alert/app 로그의 ORA- 에러
    

특히 **`parse count (failures)`가 같이 높으면**  
“parse는 많이 하는데 execute 전에 죽는다” 가능성이 큽니다.
